How it works

Every message tracked, every failure visible, every retry automatic.

Cross-channel failover - automatic, sub-second, silent

When a message fails on the primary channel, Pronto automatically escalates through a configurable cascade without any manual intervention. At the carrier level, Pronto monitors delivery rates in real-time and reroutes to a secondary carrier the moment primary performance drops below threshold.

  • Automatic cascade: SMS fails, escalates to WhatsApp, then Email
  • Carrier-level failover - sub-second reroute, transparent to recipients
  • Configurable per channel, per geography, and per message type
  • Every failover step logged with a full diagnostic trail
  • Partial failure handling - one channel failing does not block others

Event streaming - pipe every delivery event into your monitoring stack

Pronto's Event Streaming Layer (PESL) publishes platform events to 11 external destinations. When a message fails, a contact opts out, or a campaign completes, the right team gets notified in the right tool - automatically, without polling, and with cryptographic proof the payload came from Pronto.

  • 11 destination types: Webhook, AWS SQS, AWS SNS, Kafka, Microsoft Teams, Slack, Datadog, Discord, Google Chat, PagerDuty, SAP
  • 8 event types: message.sent, message.delivered, message.failed, inbound.received, contact and template lifecycle events
  • Filter by event type and by channel - only stream what each destination needs
  • HMAC signature on webhook deliveries - X-Pronto-Signature: sha256=<hmac>
  • Rate limiting, enable/disable toggle, and test delivery per connector

Common questions

Delivery and resilience, answered

When a message fails to deliver on the primary channel, Pronto automatically escalates through a configurable cascade - typically SMS to WhatsApp to Email. Failover is sub-second, transparent to recipients, and every step is logged with a full diagnostic trail. At the carrier level, Pronto monitors delivery rates in real-time and reroutes traffic to a secondary carrier if the primary falls below threshold. You can configure failover rules per channel, per geography, or per message type.
Pronto tracks delivery at two levels: per-recipient contact status (Queued, Sent, Delivered, Bounced, Failed) updated in real-time via provider callbacks from SendGrid, Twilio, and WhatsApp; and campaign-level aggregation giving you headline counts across the full send. For email, bounce classifications distinguish between hard bounces, soft bounces, dropped messages, and deferrals. All logs are filterable and exportable.
Pronto's Event Streaming Layer supports 11 destination types: Webhook, AWS SQS, AWS SNS, Kafka, Microsoft Teams, Slack, Datadog, Discord, Google Chat, PagerDuty, and SAP. You filter by event type and by channel so each destination only receives the signals it needs. Webhook destinations include HMAC signature verification using X-Pronto-Signature, so recipients can confirm every payload originated from Pronto.
Pronto retries failed delivery attempts automatically up to 5 times using exponential backoff, starting at 20 seconds. Failed jobs are retained in the queue for inspection rather than discarded. When all retry attempts are exhausted, the message is marked failed with the reason captured - and a message.failed event fires to all configured streaming destinations, so your PagerDuty, Slack, or Datadog instance is notified immediately.
