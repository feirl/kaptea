The problem

Enterprises don't have a permissions problem. They have a governance gap.

Your ops team changed a template. You found out when the complaint landed.

Without approval workflows, any user with access can edit live message templates. By the time compliance hears about it, the message has already reached thousands of customers.

Marketing sent to a segment that Compliance never cleared.

When every user sees every contact list, the wrong audience is one misclick away. Division-level guardrails don't exist, so there's nothing to stop a campaign reaching the wrong business unit's customers.

How it works

A governance layer built into every action, every user, every message.

Role-based access - three tiers, twelve modules

Pronto's permission model gives you precise control over who can do what. Superadmins have full platform access. Admins manage users, approve templates, and view all analytics. Ambassadors get operational access scoped to their modules and their division - nothing more.

  • Three roles: Superadmin, Admin, Ambassador
  • 12 granular permission modules including analytics, templates, exports, and task dashboard
  • Module-level toggles so each Ambassador is configured to exactly the right access
  • SSO via OIDC and OAuth2 - Okta, Azure AD, Google, or any generic OIDC provider
  • User attributes and division membership synced automatically from SSO claims on login
Key capabilities

Division segmentation - scoped visibility at every level

Pronto's division model mirrors the way large enterprises actually operate. Each division has its own isolated view of templates, contacts, analytics, and API keys - so a regional team in one business unit never sees data from another.

  • Division-scoped templates: Ambassadors see only their division's approved content
  • Division-scoped contacts: agents cannot access contact lists outside their scope
  • Division-scoped analytics: each business unit reports independently
  • Division-scoped API keys for programmatic access without cross-tenant risk
  • Auto-mapping from SSO department or office location attributes
Key capabilities
Each division sees only its own contacts, templates, and reports
Cross-division visibility granted at Admin level only
Usage tracked and reportable per division for billing clarity
Separate approval chains: division heads approve their own sends

Template governance - approve before it reaches a customer

Every template change made by an Ambassador goes through a change request workflow before going live. Admins review the proposed change alongside the current version, add review comments, and approve or reject. Nothing ships without sign-off.

  • Change request workflow with side-by-side comparison view
  • Review comments captured and retained as part of the approval record
  • Admin notified by email when a change request is submitted
  • Full change history: who requested, who approved, when, and what changed
  • Template categories and types scoped to division for structured organisation
Key capabilities
Workflow: Draft, Review, Approved, Archived
No template reaches a customer without passing approval
Version history: every change recorded with author and timestamp
Compliance hold: templates frozen pending regulatory review

Audit logging - every action recorded, every log exportable

Pronto logs all mutations to the platform in two complementary audit systems: request logs covering all API write operations, and template audit logs tracking the full template lifecycle. Both are searchable, filterable, and exportable as CSV for regulatory submissions.

  • Request logs capture method, path, user, IP, status, and duration for every POST, PUT, and PATCH
  • Sensitive fields redacted automatically - passwords, tokens, and API keys never appear in logs
  • Template audit events: created, change requested, approved, rejected - with timestamps and user identity
  • Filter by date range, user, action type, status code, or API path
  • CSV export accepted by ICO, FCA, and sector-specific regulators
Key capabilities
Every send, edit, login, and approval recorded automatically
Immutable log: entries cannot be edited or deleted
Exportable to CSV, JSON, or direct SIEM integration
Retention period configurable to meet your compliance policy

Control and governance, answered

Pronto has three roles: Superadmin (full platform access), Admin (template approval, user management, and full analytics), and Ambassador (operational access with 12 granular permission modules). Each Ambassador role is individually configured - you control exactly which modules they can access. Roles can be scoped per division so a user with Ambassador access in one business unit has no visibility into another.
When divisions are enabled, Ambassadors see only the templates, contacts, and analytics that belong to their division. Division-scoped API keys let each business unit integrate independently without cross-tenant data exposure. Divisions can be auto-mapped from SSO attributes such as department or office location, so your org structure in Azure AD or Okta drives access in Pronto automatically - and stays current as people move between teams.
Pronto maintains two complementary audit systems. Request logs capture all API write operations - method, path, user identity, IP address, user agent, response status, and duration - with sensitive fields like passwords and tokens automatically redacted. Template audit logs track the full template lifecycle: creation, change request submission, approval, and rejection, each with timestamp and user identity. Both are filterable and exportable as CSV in formats accepted by ICO, FCA, and other regulators.
Yes. Pronto supports SSO via OIDC and OAuth2 with Okta, Azure AD, Google, and any generic OIDC provider. User attributes - including name, division, and phone number - are mapped from your SSO claims automatically. Division membership is synchronised on every login, so joiners and leavers are reflected without manual administration in Pronto.
