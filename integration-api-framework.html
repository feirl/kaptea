Two-Way CRM Sync

Your CRM and Pronto, always in agreement

The Pronto Connector Framework maintains a persistent, bidirectional sync with your CRM. Contact records, opt-out status, custom fields, and audience segments flow both ways - automatically, in real time.

OAuth 2.0 handles authentication. An internal cursor state tracks every incremental change, so even high-volume CRMs stay fully synchronised without batch imports or manual exports.

  • Salesforce, Dynamics 365, HubSpot, Zoho CRM, SAP, Hivebrite
  • Contact create, update, and opt-out events synced in both directions
  • Custom field mapping per CRM - no schema restrictions
  • OAuth 2.0 authentication with automatic token refresh
  • Incremental sync via cursor state - no full re-imports

Event-Driven Architecture

Any event. Any channel. One API call.

Send a single POST /api/event/send-template and Pronto handles the rest. Specify the contact, the template, and the channel - Pronto resolves opt-in status, applies division rules, queues delivery via BullMQ, and reports back per channel.

The payload is fully flexible: map any field from your event schema to any template variable. No rigid schemas, no field-name restrictions.

  • Multi-channel in one call: email, SMS, WhatsApp, push
  • Per-channel success and failure response objects
  • BullMQ-backed queue for reliable, retry-safe delivery
  • Configurable key mapping - match any event schema
  • JWT Bearer + division-scoped API key authentication
// POST /api/event/send-template
{
  "contactId": "c_8af3b2",
  "templateId": "tpl_welcome_v3",
  "channel": ["email", "sms"],
  "variables": {
    "first_name": "Sarah",
    "account_ref": "ACC-2024-0091"
  }
}
// 200 Response
{
  "email": { "status": "queued" },
  "sms": { "status": "queued" }

Open APIs & Webhooks

Full programmatic control. Both directions.

Every Pronto capability is accessible via a documented REST API. Manage contacts, templates, campaigns, event streams, and user permissions - all from your own tooling, with no proprietary abstractions or workarounds.

Outbound webhooks use the Pronto Event Streaming Layer (PESL) with HMAC-SHA256 signing on every payload. Inbound callbacks from Twilio and SendGrid are handled automatically - delivery status writes back to your campaign records without extra configuration.

  • Inbound callbacks: Twilio (SMS status), SendGrid (email status)
  • OAuth 2.0 for CRM token management

Common questions

Which CRMs does Pronto support bidirectionally?
Pronto natively syncs with Salesforce, Microsoft Dynamics 365, HubSpot, Zoho CRM, and SAP. All five use the same Connector Framework, so configuration is consistent and updates are maintained by Kaptea. Hivebrite is also supported for member management use cases.
What happens if our CRM field names don't match Pronto's schema?
Field mapping is fully configurable per CRM. You define which CRM fields map to which Pronto contact properties - there are no reserved field names or schema restrictions. Custom fields are supported on both sides.
How does authentication work for the API?
All API requests require a JWT Bearer token and a division-scoped x-api-key header. CRM connectors authenticate via OAuth 2.0 with automatic token refresh. SAP uses a configuration-based auth flow.
How do we verify that incoming webhook payloads are genuine?
Every outbound webhook from Pronto includes an X-Pronto-Signature header containing an HMAC-SHA256 hash of the payload. Verify it against your shared secret to confirm authenticity before processing.
Does the send-template API support multiple channels in one call?
Yes. Pass an array of channels in the "channel" field - for example ["email", "sms"]. Pronto routes each channel independently and returns a per-channel status in the response. BullMQ queues ensure reliable delivery even under load.
What if we need an integration that isn't in the native list?
Any system that can send or receive HTTP requests can be connected via the open API or outbound webhooks. You retain full programmatic access to all Pronto resources - contacts, templates, campaigns, and event streams - without any proprietary abstraction layer.
